# 业务叙事建模（Business Narrative Modeling）

> 版本：v1.0 | 2026-04-17 产出
> 地位：**v2.3 建站流程的必做前置步骤**，所有后续技能（build-startup / content-scraping / build-execution / industry-boost / full-optimization / delivery-review）的输入依赖。
> 触发来源：2026-04-17 Demo-A v1 MVP 客户满意 vs v2 满配客户不满的反差事件。结论：模板复刻是建站大忌。

---

## 为什么要有这一步

### 我们过去的路径（错）

```
客户信息 → 选行业蓝图 → 按蓝图铺 11 个 section → 填数据 → 开功能
```

问题：
- 蓝图是"通用订菜单"，不是"这个客户的答案"
- 同一行业不同客户（Solution Integrator vs Pure Manufacturer）被塞进同一套结构
- 客户看到的是"xx 行业通用模板"，没有业务辨识度
- 功能位默认全开 → 每个功能都半成品 → 首页变"功能秀"

### 我们要走的路径（对）

```
客户信息
  ↓
① 业务模型判定（不只是行业，而是商业定位）
  ↓
② 叙事单元提取（3-5 个只有这个客户讲得出的故事）
  ↓
③ 叙事 → 页面/Section 反推（叙事决定结构，不是蓝图决定）
  ↓
④ 行业蓝图 = 参考菜单（可删可加可改，不是必用）
  ↓
⑤ 填充数据 / 开功能 / 填充视觉
```

### 一句话定性

> **蓝图是素材库，不是模具。业务叙事是骨架，模板是血肉。**

## 第一步：业务模型判定（7 选 1）

不要停留在"客户做什么产品"，要判定**客户的商业定位**。同一行业可以有完全不同的商业模型，网站叙事也完全不同。

### 7 种 B2B 业务模型

| # | 业务模型 | 核心特征 | 网站核心叙事 | 典型 section |
|---|---|---|---|---|
| 1 | **Pure Manufacturer**（纯制造商） | 自有工厂、做自己的产品、按 SKU 卖 | 产品参数 + 产能 + 工厂 + 认证 | hero / products / factory / certifications / cta |
| 2 | **Solution Integrator**（方案整合商） | 不只卖产品，卖整套方案；可能自己做一部分、整合供应商做一部分 | 按买家类型切方案 + 整合能力 + 供应商网络 | hero / **who-we-serve** / **solutions/[type]** / **supplier-network** / cases / cta |
| 3 | **OEM/ODM**（贴牌代工） | 为其他品牌生产，不做自己品牌 | 定制能力 + 客户案例 + 质量体系 + 保密承诺 | hero / capabilities / cases / quality / nda-assurance / cta |
| 4 | **Brand Owner**（品牌商） | 有自己的品牌价值主张，渠道多样 | 品牌故事 + 设计哲学 + 渠道 + 用户评价 | hero / brand-story / design / retail-network / testimonials / cta |
| 5 | **Trader**（贸易商） | 不生产，专做采购 + 出口 | 产品广度 + 供应链 + 物流 + 服务 | hero / product-categories / sourcing-network / logistics / cta |
| 6 | **Material Supplier**（材料供应商） | 卖原材料/半成品给其他 B 端制造商 | 技术指标 + 应用场景 + 批量供应 + MSDS/TDS | hero / specs / applications / technical-docs / bulk-supply / cta |
| 7 | **Service Provider**（服务商） | 卖服务不卖产品（安装/测试/咨询） | 服务流程 + 资质 + 案例 + 团队 | hero / services / process / credentials / team / cases / cta |

### 判定方法（不是猜，而是从事实推导）

问客户 5 个问题：

1. **你的收入 80% 来自"卖产品"还是"卖方案/服务"？**
   - 卖产品 → 1/3/4/5/6 里选
   - 卖方案/服务 → 2/7

2. **产品是你自己厂生产的吗？**
   - 是 → 1/3/4/6
   - 一部分是 → 2
   - 完全不生产 → 5/7

3. **你有自己的品牌吗？**
   - 有并主推 → 4
   - 有但也做别人品牌 → 1/3
   - 没有 → 5/6/7

4. **买家是谁？**
   - 终端消费者/零售商 → 4
   - 其他制造商（要用你的材料造东西）→ 6
   - 工厂（要用你的设备生产）→ 1/2
   - 品牌商（让你代工）→ 3
   - 进口商/分销商 → 5
   - 需要服务的工厂/企业 → 7

5. **卖点是什么？**
   - 产品规格领先 → 1/6
   - 整合能力（帮你搞定一堆事）→ 2
   - 代工能力（灵活、保密、稳定）→ 3
   - 品牌信任 → 4
   - 货源 + 速度 + 价格 → 5
   - 专业性 + 经验 → 7

交叉验证后输出：**"客户 X 是 [业务模型]，理由是..."**。

### 真实案例

| 客户 | 行业 | 判定 | 为什么 |
|---|---|---|---|
| Demo-A | 冷链设备 | **Solution Integrator** | 收入来自"整条冷链产线方案"；模具+机器部分外包给长三角集群；核心卖点是"给你搞定从模具到产线" |
| Demo-B | 化工胶粘剂 | **Material Supplier**（+ 轻 Trader） | 卖胶水给包装厂/家具厂；自有配方 500+；买家是下游制造商 |
| 客户A China EPS | EPS 设备 | **Pure Manufacturer** | 自己生产 EPS 机器卖给 EPS 板材厂；无整合、无代工、无品牌主张 |

**警示**：Demo-A v2.0 的错误就是**按 Pure Manufacturer 模板建的**（15 款独立产品页 + 程序化 SEO）。业务模型判错，后面再对也错。

## 第二步：叙事单元提取（3-5 个）

### 什么是叙事单元（Narrative Unit）

一个叙事单元 = **只有这个客户讲得出的故事**。判定标准三条（缺一不可）：

1. **独特性**：去掉公司名后，同行竞品不能直接搬去用
2. **有证据**：数据 / 案例 / 实物 / 工厂 / 专属内容源支撑得住
3. **有形式**：能落到具体 section 或独立页，不是虚标语

### 提取方法：三问法

问 3 遍"为什么选你不选别家"，每次深挖一层。

**第 1 问 · "你有什么优势？"** → 通常得到：质量好 / 价格好 / 服务好 → **这些都不是叙事单元**（同行都会说）

**第 2 问 · "为什么你的质量/价格/服务比别家好？"** → 通常得到：我们有 XX 证书 / XX 年经验 / XX 技术 → **接近但还不够**（证书人人有）

**第 3 问 · "这个 XX 证书/经验/技术，具体帮客户解决了什么别家解决不了的问题？"** → 这里才是**叙事单元的金矿**

### 真实案例：Demo-A 的三个叙事单元

| # | 叙事单元 | 独特性 | 证据 | 承载形式 |
|---|---|---|---|---|
| 1 | **Who We Serve By Factory Type** | "我们按你的工厂类型给方案"而非"按产品卖货" | 冰箱厂/冷柜厂/冷库厂/冷藏柜厂/饮料冷柜厂/酒窖制冷厂 6 种切片 | 首页 Who-We-Serve section + `/solutions/[type]` 独立页 × 6 |
| 2 | **Yangtze-Delta Supplier Cluster** | 长三角产业集群出 80% 中国冷链模具 | 地理事实 + 合作供应商清单 | 独立页 `/factory-network` |
| 3 | **Turnkey, Not Piece-by-Piece** | 一家对接 vs 5 家分别谈 | 案例对比数据（采购周期 / 沟通成本 / 品控一致性） | 首页 ValueProp section + 案例佐证 |

### 叙事单元禁忌

❌ **不要选**：
- "高品质产品"（虚标语）
- "专业团队"（人人有）
- "30 年经验"（是数字不是故事，只能做 TrustBar 不能做叙事单元）
- "客户遍布全球"（人人说）

✅ **要选**：
- "为你解决 XX 具体问题的 XX 具体能力，证据是 XX"
- 能拿给同行看，同行会说"我做不到"的
- 客户自己讲出来会有自豪感的，不是话术

### 数量

- **3 个起步，5 个封顶**
- 少于 3 个：业务辨识度不足，网站会变通用模板
- 多于 5 个：注意力稀释，没有一个能讲深

## 第三步：叙事 → 页面/Section 反推

### 反推规则

每个叙事单元必须回答三个问题：

1. **叙事形式**：这是"一块 section 能装下"还是"需要独立页"？
2. **section 类型**：首页哪一屏？独立页叫什么名字？
3. **匹配现有 variant 吗**：27 个现有 variant 里有合适的吗？没有就**新造**，不要硬塞。

### 反推示例（Demo-A）

```
叙事 1: Who We Serve By Factory Type
  ├─ 首页 section：Who-We-Serve（6 卡片，按工厂类型）
  └─ 独立页：/solutions/[slug] × 6（每种工厂类型一页，深讲方案）
       → 现有 variant 无匹配，新造 solutions-grid variant

叙事 2: Yangtze-Delta Supplier Cluster
  └─ 独立页：/factory-network
       → 现有 variant 无匹配，全新页
       → 首页不单独占 section（信息密度太重，独立页承载）

叙事 3: Turnkey, Not Piece-by-Piece
  └─ 首页 section：ValueProp（4 格编号卡片）
       → 现有 variant 有：value-prop-numbered
```

### 最终页面地图（由叙事反推得出）

```
首页：Hero / TrustBar / 【叙事1: Who-We-Serve】/ 【叙事3: ValueProp】/ Products / CTA （6 section）
独立页：
  /solutions/[type] × 6（叙事1 落地）
  /factory-network（叙事2 落地）
  /products（产品类目组聚合，2-4 组）
  /about /contact /cases
```

**对比蓝图默认做法**：蓝图会塞 11 个 section（hero / trust-bar / logos / stats / products-grid / advantages / factory / process / testimonials / faq / cta），其中 7 个对 Demo-A 业务无独特性 → 堆砌。

## 第四步：行业蓝图降级使用

### 蓝图的新地位

- **以前**：`layout: blueprints['cold-chain']` = 整套 11 section 强制套用
- **现在**：蓝图是"同行业常见 section 参考清单"，用法：
  1. 看蓝图的 `suggestedHome` 列表
  2. 每条问："这个 section 能承载我提取的叙事单元吗？"
  3. 能承载 → 保留
  4. 不能承载 → 删除
  5. 叙事需要但蓝图没有 → 新造

### 蓝图审查 5 问

接到一个客户，拿到蓝图建议列表后，对每个 section 问：

1. 这个 section 在讲哪个叙事单元？
2. 它讲的是**这个客户独有的**故事还是**行业通用**故事？
3. 我们有**真实素材**填充吗（不是编的、不是占位、不是 emoji）？
4. 删掉这个 section，网站故事还完整吗？（完整 → 应删 / 不完整 → 保留）
5. 这个 section 在**客户的业务模型**里是不是必需（参考第一步 7 种模型的典型 section 表）？

## 第五步：反模板复刻自检 5 问

**建站完成前，智能体对自己灵魂拷问**：

1. **去掉公司名和 logo，同行业其他客户能直接套用这个网站吗？**
   - 能 = 错了。回去做叙事建模。
   - 不能 = 过关。

2. **客户看完首页能用一句话说出"你和同行有什么不同"吗？**
   - 说不出 = 错了。叙事单元没有落地到首页。
   - 说得出 = 过关。

3. **砍掉一半 section，网站故事还成立吗？**
   - 不成立 = 本来就有废料，把废料删了。
   - 成立 = 那些真的是废料，确实该删。

4. **我在填模板还是在讲故事？**
   - 看 data/*.ts 数据有多少是真实的、多少是占位的。占位 > 30% = 错了。

5. **我的产出比手搓 MVP 版本讲得更清楚吗？**
   - 不清楚 = 负优化。满配反而稀释了叙事。
   - 清楚 = 过关。

## 交付物：叙事档案

业务叙事建模的产出 = **《{客户}业务叙事档案.md》**

模板见：[叙事档案模板.md](叙事档案模板.md)

**强制要求**：
- 放在客户项目根目录：`客户/独立站-客户XXX/业务叙事档案.md`
- build-startup 完成前必须产出
- build-execution / content-scraping / industry-boost 必须读它、据它决策
- delivery-review 必须对照它做交付审核

## 与 v2.3 其他规则的关系

| 规则 | 关系 |
|---|---|
| DNA 满配（5 Schema / 12 GA4 / 站内 AI 等） | **保持满配**（DNA 是技术基建不是业务叙事）|
| 27 section variants | **保持存在，但不默认套用**（素材库角色）|
| 7 套行业蓝图 | **降级为参考菜单**（suggestedHome，不是必须）|
| 功能层开关（wizard / ROI / 程序化 SEO 等）| **默认全关**，按业务模型逐个开|
| astro-b2b-starter fork 起步 | **保持**（技术起点不变）|
| 9 技能流程 | **保持**，但 build-startup 新增"叙事建模"必做前置步骤|

## 后续沉淀

每建一个客户 → 叙事档案归档 → **经验库/业务模型库/{业务模型}.md** 积累该模型的典型叙事单元 / section 组合 / 常见陷阱。

7 种业务模型 × 7 种行业 = 49 种组合。每组合积累 3-5 个案例后，智能体对新客户的判定准确度会指数级上升。

**这是长期复利，不是一次性工程**。
