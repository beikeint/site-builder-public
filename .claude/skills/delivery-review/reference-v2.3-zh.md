# 交付验收技能 v2.4 — 含去名称压力测试的终极防线

> 适用场景：网站开发完成后，从竞品对标、质量验收、文档编写到最终上线的完整交付流程。
> 使用者：独立站建站智能体（内部）
> 更新时间：2026-04（v2.3）→ **v2.4 升级**

---

## 🚨 v2.4 升级补丁

### 新增：去名称压力测试（终极防线）

交付前最后一道关卡。**防止 Demo-A v2 式"看起来满、实际像模板"的交付重现**。

### 测试方法

1. **截图首页**（1920×3000 全屏截图）
2. **PS/Figma 去掉**：
   - 公司 logo
   - 公司名（Hero 标题里的、Footer 里的、TrustBar 里的）
   - 联系方式（WhatsApp / 电话 / 邮箱）
   - 地址
3. **把去名版截图发给用户（运营人员）**，问两个问题：
   - Q1：**"这是哪家公司的网站？"**
     - 用户说不出 → ✅ 通过（真差异化）
     - 用户能说出 → ❌ **过度依赖名称信任建设，回炉加叙事单元**
   - Q2：**"这个网站给同行业其他客户用合适吗？"**
     - 用户说"不合适，这是给做 XX 的"→ ✅ 通过（业务辨识度强）
     - 用户说"挺通用的，好像都能用"→ ❌ **模板复刻陷阱，回炉**
4. **两问都过 → 允许交付**，单问不过 → 回炉

### 反模板复刻交付前自检 5 问（必答）

- [ ] 1. 去掉公司名和 logo，同行业其他客户能直接套用这个网站吗？**必答"不能"**
- [ ] 2. 客户看完首页能用一句话说出"你和同行有什么不同"吗？**必答"能"**
- [ ] 3. 砍掉一半 section，网站故事还成立吗？**不成立 = 砍**
- [ ] 4. `data/*.ts` 占位数据占比 < 30%？
- [ ] 5. 产出比手搓 MVP 版本讲得更清楚吗？**不清楚 = 负优化回炉**

### 交付文档新增必附

- [ ] 《业务叙事档案.md》完整版（归档 + 随交付给客户一份）
- [ ] 去名称压力测试结果截图 + 用户答复记录
- [ ] MVP 对照测试结论（来自 full-optimization）

### 五维度评分表升级（v2.4）

原五维度（技术 / SEO / 转化 / 视觉 / 内容）+ 新增第六维度：**业务叙事一致性**

| 维度 | v2.3 权重 | v2.4 权重 |
|---|---|---|
| 技术 | 20% | 15% |
| SEO | 20% | 15% |
| 转化 | 20% | 15% |
| 视觉 | 20% | 15% |
| 内容 | 20% | 15% |
| **业务叙事一致性** | — | **25%**（新增，权重最高）|

业务叙事一致性评分不足 80 分 → **一票否决**，不允许交付。

### 硬规则（v2.4）

- ❌ **禁止未过去名称压力测试就交付**
- ❌ **禁止交付物里无业务叙事档案**
- ❌ **禁止业务叙事一致性评分 < 80 分放行**
- ❌ **禁止同一错误（如模板复刻陷阱）第二次出现在交付物中**（经验库有记录仍犯错 = 严重失误）

### 对照失败案例

- ❌ **Demo-A v2**：五维度全通过（页数多、Schema 齐、GA4 接入、图有 alt、SEO 合规），但**去名称测试必败**——同行业其他冷链设备商可直接套用这个网站 → 客户反馈乱、草稿、拼贴
- ✅ **Demo-A v1 MVP**：五维度不全（仅 115 页、程序化 SEO 无、部分 Schema 缺），但**去名称测试必过**——Who-We-Serve / Factory-Network / Solutions 三大叙事单元是 Demo-A 独有 → 客户满意

**结论**：**叙事一致性压倒技术满配**。v2.4 正式把叙事一致性升级为交付第一指标。

- 网站开发完成，准备进入交付阶段
- 需要对网站做上线前的全面检查
- 需要生成客户交付文档（交付说明书、使用指南）
- 需要与竞品网站做对标分析，量化自身水平

## 二、交付流程（4步）

```
第1步：竞品分析 → 知道我们在什么水平
第2步：阶段验收 → 五维度打分，找出短板
第3步：上线检查 → 逐项核查，确保零遗漏
第4步：交付文档 → 给客户一份完整的使用说明
```

## 三、核心提示词

### 提示词1：竞品分析报告

```
请对以下网站进行竞品对标分析：

我们的网站：【URL或项目路径】
参考站（同行）：【URL】
行业标杆：【URL】

分析维度（6个），每个维度用表格对比：

1. 询盘转化设计
   对比：浮动按钮、CTA数量、产品页询盘入口、信任标签

2. 产品展示
   对比：图片质量、分类方式、详情页深度、SKU数量

3. 信任背书
   对比：认证展示、数据展示、客户案例、视频内容

4. SEO & 内容
   对比：Title优化、FAQ、博客/文章、Schema、Sitemap

5. 视觉设计
   对比：整体风格、配色、动效、移动端

6. 技术
   对比：图片格式、CSS/JS体积、页面加载

每个对比点标注：✅ 有 / ❌ 无，以及 "我们优于/持平/弱于" 的判断。

输出：
- 逐维度对比表格
- 我们的核心优势（Top 5）
- 我们的主要差距（按优先级排序，含改进建议）
- 结论：一段话总结
```

### 提示词2：五维度阶段验收报告

```
请对当前网站进行五维度阶段验收，输出正式的验收报告。

验收标准：跨境B2B询盘独立站行业标准

验收维度及评分规则：

一、技术（满分10分）
得分项（每项+1分）：
- 静态站性能（CSS+JS < 50KB）
- 构建零错误
- Sitemap自动生成
- robots.txt配置
- 移动端适配
- .htaccess（HTTPS+重定向）
- Canonical URL
- 每页唯一H1
- WebP图片
- hreflang标签

二、SEO（满分10分）
得分项：
- 每页独立title+description（+2）
- Product Schema（+1.5）
- FAQ Schema（+1.5）
- 图片alt文字（+1）
- URL结构简洁（+1）
- OG标签完整（+1）
- 内链结构（+0.5）
- hreflang（+0.5）
- 博客/文章（+0.5）
- llms.txt（+0.5）

三、转化（满分10分）
得分项：
- 浮动询盘按钮（+2）
- Hero CTA（+1）
- 导航栏CTA（+1）
- 产品页双入口（+1）
- 信任标签（+1）
- 表单预填（+0.5）
- Hero信任行（+0.5）
- CTA Banner（+0.5）
- 感谢页+GA4（+0.5）
- WhatsApp（+0.5）
- 客户案例（+0.5）

四、视觉（满分10分）
得分项：
- 整体风格一致（+2）
- Hero设计（+1.5）
- 认证信任条（+1）
- 卡片交互（+1）
- 移动端视觉（+1）
- 页面一致性（+0.5）
扣分项：
- 产品图质量差（-1）
- 缺工厂照片（-0.5）
- 缺证书展示（-0.5）
- 无视频（-0.5）

五、内容（满分10分）
得分项：
- 产品数据完整（+1.5）
- 公司介绍（+1）
- FAQ内容（+1）
- Quality页面（+1）
- Privacy Policy（+0.5）
- 联系信息（+0.5）
- 博客文章（+1）
- 多语言翻译完整（+1.5）
- 客户案例（+0.5）
- 产品描述丰富（+0.5）

六、博客文章质量验收（每篇文章单独评分，低于7分必须修改后才能交付）

每篇博客文章按以下4个维度打分，每个维度满分10分，取平均分为文章总分。

6.1 SEO维度（满分10分）
- 目标关键词在标题中出现（+1.5）
- 目标关键词在首段出现（+1）
- 目标关键词在至少1个H2中出现（+1）
- 文章长度1500词以上（+1.5）
- 有2-3个产品页内链（+1.5）
- 所有内链有效无404（+1）
- 图片有alt文字且含关键词（+1）
- Article Schema结构化数据完整（+1）

6.2 GEO维度（满分10分）
- 开头有清晰的问题定义（+2.5）
- 有FAQ部分且至少5个问题（+2.5）
- FAQPage Schema已配置（+2）
- 有具体数据支撑观点（每150-200词一个数据点）（+3）

6.3 准确性维度（满分10分）
- 公司认证信息与网站一致，不多写（+2.5）
- 产品参数与网站产品页一致（+2.5）
- 数字和数据有来源或限定词（+2.5）
- 竞品信息用保守表述（+2.5）

6.4 可读性维度（满分10分）
- 段落结构清晰，有H2/H3层级（+2）
- 有对比表格辅助理解（+2）
- 有列表/要点归纳（+2）
- 语言专业但易懂，适合B2B采购商阅读（+2）
- 结尾有CTA引导询盘（+2）

评分判定规则：
- 文章总分 ≥ 7分 → 通过验收，可交付
- 文章总分 < 7分 → 必须修改后重新评分，不得交付
- 任何单一维度 < 5分 → 该维度为严重不合格，必须优先修改

输出格式：
- 总评分（5维度平均）
- 每个维度：得分 + 评级 + 得分项表格 + 扣分项表格
- 待改进清单：必须修复（低于8分的维度） + 建议改进
- 结论（一段话）
```

### 提示词2.5：交付前补强执行（上线检查之前必须完成）

```
请在上线检查之前，执行以下7项补强任务，确保网站达到行业顶级水准。

一、补全行业标准缺口（5项检查）

逐项检查并补全：
1. 认证展示完整性 — 所有认证（CE/ISO/FDA/BSCI等）在认证页面有独立条目（图标+名称+编号+适用标准），产品详情页对应产品显示相关认证标签
2. 产品规格完整性 — 每个产品至少8个规格参数（材质/尺寸/重量/颜色/标准/包装/MOQ/认证），缺失的标注"Contact for details"
3. 信任数据具体化 — Why Us的4个数据不能用模糊词（"many"/"large"），必须用具体数字（"15+ Years"/"200M+ Units/Year"/"80+ Countries"/"500+ Clients"）
4. 询盘入口数量 — 确保至少5个入口：导航栏CTA、Hero区CTA、产品页Send Inquiry、浮动按钮、CTA Banner。如果少于5个，补齐
5. Privacy Policy — 必须有隐私政策页面，多语言版本，包含GDPR合规条款

二、B2B最佳实践强化清单

逐项执行：
- [ ] Hero信任行存在且包含4项（Free Samples / Low MOQ / 24h Reply / OEM Available）
- [ ] 认证信任条在Hero下方（5-6个认证图标一行排列）
- [ ] 产品卡片有Bestseller/New角标
- [ ] 下单流程4步图示完整（Inquiry → Sample → Order → Delivery）
- [ ] CTA Banner在页面中段（渐变背景 + 行动号召 + 按钮）
- [ ] 产品详情页有相关产品推荐（同分类4个产品）
- [ ] 产品详情页有面包屑导航
- [ ] About页面有公司里程碑/发展历程
- [ ] Contact页面有地图嵌入或地址图示
- [ ] 页脚包含公司全称+地址+"Verified Supplier since [年份]"

三、移动端专项优化

用移动端视角检查并修复：
1. 导航菜单：汉堡菜单展开后，语言切换和CTA按钮都能正常操作
2. Hero区：标题不超过2行，CTA按钮足够大（最小48x48px触控区域）
3. 产品卡片：单列排列，图片清晰，信息不截断
4. 表单：输入框宽度100%，键盘弹出后表单不被遮挡
5. WhatsApp按钮：不遮挡页面内容，不与浮动询盘按钮重叠
6. 表格：产品规格表格水平可滚动，不撑破布局
7. 字体：最小14px，行高1.6，阅读舒适
8. 退出意图弹窗：移动端已禁用

四、极限速度优化

执行以下性能优化：
1. 图片全部WebP格式，质量80%，宽度不超过1200px
2. CSS总体积 < 50KB（移除未使用的Tailwind类）
3. JS总体积尽量为0（Astro静态站优势），必要JS延迟加载
4. 首屏图片预加载（Hero背景图/产品图），非首屏图片lazy loading
5. 字体：系统字体优先，如有自定义字体用font-display: swap
6. 总打包体积 < 3MB（不含图片外链情况下）
7. 检查是否有未压缩的大文件

五、结构化数据增强

检查并补全以下5种Schema：
1. Organization（首页）：name, url, logo, contactPoint, address, sameAs, foundingDate
2. Product（每个产品详情页）：name, description, image, brand, manufacturer, offers, sku, aggregateRating（如有）
3. FAQPage（首页FAQ区域）：mainEntity包含所有Q&A
4. Article（每篇博客）：headline, author, datePublished, dateModified, image, publisher
5. BreadcrumbList（产品详情页）：itemListElement完整面包屑链
- 用JSON-LD格式嵌入<head>
- 验证：用Google Rich Results Test工具格式检查（或人工审查JSON结构）

六、内链优化

构建完整的内链网络：
1. 首页 → 每个产品分类页（通过分类卡片）
2. 分类页 → 每个产品详情页
3. 产品详情页 → 同分类相关产品（4个推荐）
4. 产品详情页 → 返回分类页（面包屑）
5. 博客文章 → 相关产品详情页（文章内自然链接）
6. 博客文章 → 相关博客文章（底部推荐阅读）
7. 所有页面 → 联系页面（通过导航和CTA）
8. 检查：不存在孤立页面（每个页面至少被2个其他页面链接）

七、最终重新打包

完成以上所有补强后：
1. npm run build — 必须零错误零警告
2. 检查dist/目录完整性
3. 对比补强前后：页面数是否增加、体积变化、新增Schema数量
4. 输出补强报告：完成了哪些项、跳过了哪些项（及原因）
```

### 6.4 QA必检清单（12项，每次交付+每次修改联系方式/图片时必做）

> 来源：客户B/Demo-B/demo-a/demo-c 实战教训，每项都曾导致客户问题。**逐项检查，不允许跳过。**
> 升级历史：9项（客户B首次建站）→ 10项（2026-04-17 Demo-B导航死链）→ 11项（2026-04-19 Demo-B产品字段空值）→ 12项（2026-04-19 demo-c Yandex 验证失败）

```
□ 1. WhatsApp国际区号 — grep所有 wa.me/ 链接，格式必须 wa.me/86XXXXX（中国）
     ❌ 常见错误：wa.me/+86-xxx-xxxx-xxxx → 被识别为美国号码+1
     ✅ 正确格式：wa.me/86+86-xxx-xxxx-xxxx

□ 2. 电话 tel: 与显示文字一致 — grep所有 tel: href，对比页面上显示的号码
     ❌ 常见错误：tel:+86+86-xxx-xxxx-xxxx 但显示 0086+86-xxx-xxxx-xxxx（两个不同号码）
     ✅ 验证方法：href数字和显示文字必须完全相同，来源以 client-manager 为准

□ 3. 产品图片无竞品水印 — 逐张目视检查所有产品图
     ❌ 常见错误：图片带 EPSOLE/其他品牌 logo水印
     ✅ 验证方法：打开每张产品图，检查角落/中心是否有第三方文字或logo

□ 4. mailto: 有剪贴板复制兜底 — 检查所有邮箱链接点击行为
     ❌ 常见错误：海外用户无本地邮件客户端，点 mailto: 无反应
     ✅ 验证方法：所有 mailto: 按钮必须有 navigator.clipboard.writeText() + "Copied!" 视觉反馈

□ 5. 联系方式统一来源 — grep所有联系信息，确认来源
     ❌ 常见错误：不同页面硬编码了不同的电话/邮箱
     ✅ 验证方法：所有联系信息从 site.config.ts 或 i18n 翻译文件读取，禁止硬编码

□ 6. Schema结构化数据完整 — 用 seo-checker.check_structured_data 验证
     Product: name + image + offers.price + brand ← 缺任一 → ❌
     LocalBusiness: name + image + address + telephone ← 缺任一 → ❌
     （已在6.2 DNA层验收中覆盖，此处交叉确认必填字段）

□ 7. GA4转化追踪全覆盖 — 浏览器控制台逐个触发验证
     5个必配事件：whatsapp_click / email_click / phone_click / quote_click / generate_lead
     ✅ 验证方法：Network面板过滤 collect?，逐个触发确认事件发送
     （已在6.2 DNA层验收中覆盖，此处确认5个事件名正确）

□ 8. IndexNow部署 — 检查 public/ 目录+脚本
     ✅ public/ 下有 IndexNow key 文件（32位hex字符串.txt）
     ✅ scripts/index-boost.mjs 存在且可运行
     ✅ 部署后运行一次提交所有URL到 Bing/Yandex

□ 9. 表单实际提交测试 — 浏览器中提交，确认邮箱收到
     ❌ 常见错误：Web3Forms access_key 是占位符 YOUR_WEB3FORMS_ACCESS_KEY，从不报错但100%丢失询盘
     ✅ 验证方法：
        - grep -r "YOUR_" --include="*.astro" 排除所有占位符
        - 在浏览器中实际提交一次测试表单 → 确认接收邮箱收到
        - 每个语种的contact页都要测（不能只测英文）
        - Exit-Intent弹窗单独测（有独立key引用）

□ 10. 导航项必有落地页（2026-04-17 Demo-B教训）— site.config.ts 里每个 features.xxx: true 必须有对应落地页
     ❌ 常见错误：features.applications: true 但没建 /en/applications/ 页 → 全局导航死链，GSC 直接报错
     ✅ 验证方法：
        - grep `features:` 每个 true 的开关都要 grep 对应 `/src/pages/[locale]/xxx` 页面文件存在
        - 或在 Header.astro 加 fallback 打印 warning
        - 构建后 `grep -r "href=\"/[^\"]*\"" dist/` 抽查导航链接全部可达

□ 11. 产品详情页动态字段空值检测（2026-04-19 Demo-B教训）— 产品 model/price/shelfLife 等可选字段空时禁止渲染出空括号
     ❌ 常见错误：模板硬编码 `(${product.model})` → model 空时渲染 "()" 空括号；答案胶囊 / 询盘标题 / alt / 表单 subject / WhatsApp 文案 6 处全崩
     ✅ 验证方法：
        - 建站时先统计 products.ts 里 model/price/shelfLife 等字段空值率
        - 模板里禁止直接硬编码 `(${product.model})`，必须用 helper 变量 `modelLabel = product.model ? ` (${product.model})` : ''` 兜底
        - QA 抽查 fetch 1 个 model 空 + 1 个 model 非空 的产品详情页，grep `()` 空括号和 `for $` 末尾缺名
        - 覆盖所有布局变体：Standard / FullWidth / Sidebar 三个组件都要修

□ 12. 根 URL 跳转空壳必须含 verification meta（2026-04-19 demo-c 教训）— 根 URL 必须保留 google/bing/yandex 验证 meta
     ❌ 常见错误：用 `Astro.redirect(`/${defaultLocale}`)` → 静态构建降级为 302 / meta refresh 空壳，Yandex 只爬根 URL 不跟 refresh → Search Console 报"未找到元标签"
     ✅ 验证方法：
        - 建站完成后 `curl -s https://<域名>/ | grep -i "verification"` 确认根页 HTML 含 google/bing/yandex meta
        - index.astro 必须是手写 HTML 模板（不能用 Astro.redirect），verification 常量内联注入（与 BaseLayout 同步）
        - Search Console 验证必先 curl 根 URL 确认 meta 存在，再去平台点验证，避免浪费客户时间
        - starter 模板 v2.4+ 已默认带 verification 三字段和条件渲染，新建站直接继承
```

**全部通过才算交付完成。任何一项失败必须修复后重检。**
