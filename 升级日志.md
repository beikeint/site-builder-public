## v2.5.1 · starter hreflang + Schema 字段补强（2026-04-26）

**触发事件**：运营人员一句"抄 starlight hreflang + aimeos schema 给 starter"。三路并行调研后**结论反转**：

- starter 现有 hreflang **比 Starlight 更全**（Starlight 自己都没做 x-default + self-reference，starter 已有）
- Aimeos 自己用 **microdata 不是 JSON-LD**，字段完整度还不如 starter，**不该抄**
- 真正的 gap 是 5 项小坑（sitemap 互链 / Product 字段 / Org B2B 字段 / Breadcrumb @id / 判空守护），其中"判空守护"是从 Starlight 经验吸收

**5 项升级**（Batch 1 快赢，全向后兼容）：

| # | 文件 | 改动 | 验证（dist 输出） |
|---|---|---|---|
| 1 | `astro.config.mjs` | sitemap integration 加 `i18n` 参数（`{defaultLocale, locales: code→hreflang 映射}`） | `dist/sitemap-0.xml` 每条 `<url>` 自动带 `<xhtml:link rel="alternate" hreflang>` 互链 |
| 2 | `data/products.ts` + `utils/schema.ts` | Product 加 6 字段（sku/mpn/gtin/priceValidUntil/itemCondition/material/weight/aggregateRating），全可选；offers 默认带 `itemCondition: NewCondition` + `seller: Organization` | 产品页 JSON-LD `category`/`offers.itemCondition`/`offers.seller` 默认输出 |
| 3 | `config/site.config.ts` + `BaseLayout.astro` | Org Schema 多 contactPoint 数组（sales/support/technical 按填了哪个就出哪个）+ `foundingDate` + `numberOfEmployees`（QuantitativeValue） | `contactPoint: [{contactType: 'sales'}]`（默认只 sales，填了 supportPhone/technicalPhone 会自动追加） |
| 4 | `utils/schema.ts` | BreadcrumbList 每个 ListItem 加 `@id`（=item URL，搜索引擎实体识别更稳） | 每条 ListItem 都有 `@id` |
| 5 | `BaseLayout.astro` | `hasRealSiteUrl` 守护（`example.com` 视为未配置）→ canonical/og:url/hreflang 全跳过；OG image 也跟随守护 | 模板默认配置（`example.com`）下 dist HTML 无 canonical 标签，避免静默 SEO 失效 |

**调研产出（已验证有效的反结论）**：

- "Aimeos schema 完整" = 反向认知。Aimeos 用微格式（itemscope/itemprop），缺 mpn/gtin/aggregateRating；该参考的是 Google Merchant 规范 + Shopify 实现
- "Starlight hreflang 范本" = 部分。Starlight 没做 x-default 和 self-reference（B2B 场景必备）；可借鉴的只是 `localizedUrl()` 路径替换抽象 + 判空守护 + 404 不输出 hreflang

**Batch 2 待讨论**（涉及数据约定，影响所有客户站，需运营人员确认再做）：

- 翻译缺失的 hreflang 策略 — 给 product/blog data 加 `availableLocales` 字段，未翻译的 locale 不出 hreflang
- 404 / 根路径 hreflang 处理一致性
- AggregateRating 数据来源约定（无真实评论严禁伪造，是不是从询盘/Google reviews 反向接入）

**沉淀**：

- 经验库 `通用教训/结构化数据.md` 追加 v2.5.1 字段约定段
- 经验库 `通用教训/canonical-hreflang-判空守护.md` 新建（跨网站类型可复用）
- 本日志（你正在读）

## 新增技能 v2.5-visual-retrofit · 老站对齐前沿视觉 12 步 SOP（2026-04-26）

**触发事件**：v2.5 starter 升级（昨日 commit `c28c290`）后，老客户站需要批量对齐。运营人员问"客户 001 Demo-D / 客户 004 Demo-B的升级流程是什么"→ 决定把流程沉淀成可复用 skill，避免每个老站升级都靠记忆/口述。

**为什么需要单独 skill**（不是放在 build-execution 里）：

- 老站升级 ≠ 新建站。老站有上线流量、有客户改过的 Header/global.css、有已索引 URL 不能动。
- 步骤 3「流量页保护」是老站独有（v2.4 硬规则来源 = 客户 004 PVA blog 误删事件），新建站没有这个前置
- 步骤 5「Header/global.css 合并」≠ 步骤 4「core 组件直接覆盖」，新手很容易整目录 cp 然后毁了客户改过的 nav。skill 把这俩步骤拆开标红
- 后续可能扩展 v2.6 / v2.7 视觉升级，每代一个 retrofit skill 比往 build-execution 不断塞更清晰

**Skill 结构**（4 阶段 12 步）：

| 阶段 | 步骤 | 关键事项 |
|---|---|---|
| A 预检 | 1-3 | site-monitor / 当前在哪一代判定 / GSC 流量页 baseline（v2.4 硬规则） |
| B 替换 6 件套 | 4-8 | core 4 组件直接覆盖 / Header + global.css 合并 / BaseLayout 注入 / 删 widgets + pages 旧 import |
| C 适配 | 9-11 | i18n 9 key 全语种填齐 / 品牌色 token 检查 / features.exitIntent |
| D 验收发布 | 12 | npm run qa #16/#17/#18 / Playwright 7 截图 / deployer / site-monitor / client-manager 时间线 / commit / sync release |

**已沉淀的硬规则**：

- 6 个常见坑（直接 cp 整目录 / i18n 漏语种 / 品牌色没设 / 没删 widgets / 动了流量页 URL / 002/002u 已 ops 补丁要 diff）
- 已升级清单表格（含 6 个客户当前状态占位）

## 交付文档中心（2026-04-17）

**事实**：在 `智能体/建站/独立站建站-site-builder/` 下新建 `交付文档/` 作为客户相关文档的统一入口。

**Why**：demo-a 项目交付时发现需要给客户写详细的 4-Key 教程（Web3Forms / GA4 / GSC / IndexNow），这份教程可复用到每个新客户。运营人员要**"方便查看"**所有客户的交付状态和操作流程，不想翻多层目录找。

### 目录结构约定

- **智能体工作目录** `智能体/建站/独立站建站-site-builder/` — 只留智能体自身资产：CLAUDE.md + 交付文档/ + 速查卡/资料收集表 + 案例库/ + 升级日志.md
- **客户项目资料**全部在 `客户/独立站-客户XXX/`：website/ + 原始素材/ + 归档-启动期/ + 各种交付 md
- **新客户项目目录必须和 client-manager ID 对齐**（例：client-D → Demo-B-client-D，不能命名错位）

### How to apply

| 场景 | 看什么 |
|---|---|
| 新客户建完站 | 复制 `交付文档/客户操作教程-通用模板.md` 到 `客户/独立站-XXX/docs/`，替换 `{{BRAND}}` / `{{DOMAIN}}` 等占位符后发客户 |
| 想查所有客户状态 | `交付文档/已交付客户速查表.md`（表格） |
| 日常运维/紧急处理 | `交付文档/上线后运维SOP.md`（每日巡检 / 每周节奏 / runbook） |
| 场景导航 | `交付文档/README.md` — 按 5 场景（开工/交付/运营/查状态/答疑）找文档 |

CLAUDE.md 底部已有"工作区文件导航"段链入，不用记忆路径。

## P0-P1 建站阶段基础设施（2026-04-12）

**摘录自网站运营-web-ops P0-P4 天花板升级** — 建站阶段要交付的两项硬件基础设施：

### P0 · 转化追踪闭环（建站必交付）

- 全站 5 个 GA4 事件：`whatsapp_click` · `email_click` · `phone_click` · `quote_click` · `exit_intent generate_lead`
- 事件委托在 BaseLayout — 一次部署全站生效
- **QA 必检清单第 7 项**就是这个

### P1 · 索引加速引擎（建站必部署）

- IndexNow API key 部署到 `public/`
- `scripts/index-boost.mjs` 批量提交 Bing/Yandex
- **QA 必检清单第 8 项**就是这个

### P3 · ROI 计算器模板（建站可选）

若客户是 B2B 设备类 → 套 `ROI 计算器`模板到 `/en/roi-calculator/`

完整 P0-P4 体系见 [网站运营-web-ops/升级日志.md](../../运营/网站运营-web-ops/升级日志.md)。
